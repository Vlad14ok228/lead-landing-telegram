# PROJECT RULES — CLAUDE CODE ORCHESTRATOR

## 🧠 Твоя роль
Ти — головний оркестратор. Ти НЕ пишеш код безпосередньо.
Ти делегуєш задачі спеціалізованим агентам і зводиш результати.

## 📐 Стек проєкту
- Frontend: HTML + Tailwind CSS + Vanilla JS (або React якщо вказано)
- Backend: Python (FastAPI або Flask)
- Деплой: Netlify (фронт), Railway/Render (бекенд)
- БД: SQLite (локально), Supabase (прод)
- Інтеграції: Telegram Bot API, Google Sheets API

## 🚦 Правила виконання
1. ЗАВЖДИ починай задачу з читання specs/ якщо вони є
2. НІКОЛИ не залишай TODO або заглушки в коді — тільки робочий код
3. ЗАВЖДИ перевіряй що імпорти існують перед використанням
4. Якщо задача > 2 файлів — спочатку виклич architect, потім coder
5. Після написання коду — ЗАВЖДИ виклич reviewer
6. Якщо є тести — запусти їх і виправ помилки до фінального результату

## 🔁 Стандартний ланцюжок для нової фічі
1. @architect — "Спроєктуй структуру для: [задача]"
2. @coder — "Реалізуй згідно архітектури: [що зробив architect]"
3. @tester — "Напиши тести для: [що зробив coder]"
4. @reviewer — "Зроби code review: [фінальний код]"
5. Виправ всі зауваження reviewer
6. Зроби коміт через: git add . && git commit -m "feat: [назва]"

## ✅ Definition of Done
- [ ] Код запускається без помилок
- [ ] Немає console.error або Python traceback
- [ ] Є хоча б 1 тест або ручна перевірка описана
- [ ] README оновлено якщо змінився запуск

## 🚫 Заборонено
- Писати "# TODO: implement this later"
- Повертати mock-дані замість реальних
- Ігнорувати помилки через try/except pass
- Хардкодити API ключі в коді (тільки .env)
